# Corpus audit — Phase F

Institutional memory for the full-corpus source-integrity audit
launched as Phase F of BACKLOG #19. Document is authored in-progress:
Tier 1 complete as of 2026-04-24; Tiers 2–6 pending. Final closeout
will include aggregate findings, convention decisions, and a single
record of what the audit was worth across the full corpus.

---

## Why this audit exists

Phase B of BACKLOG #19 removed the `Verified` marker from rendered
node bodies and refactored the verbatim-quote check in `validate.py`
to run unconditionally on every block-quote with a Source row. The
principle from Phase E codified the rule: confirmation against the
underlying primary source is a precondition for inclusion, not a
rendered flag.

But the 508 existing quotes in the corpus were all verified under
the OLD regime, which substring-matched quote text against pdftotext
output of the cited source. When the PDF's text layer faithfully
represents the original (text-native sources), pdftotext output IS a
reasonable proxy for the original and the check has teeth. When the
text layer is unreliable — OCR'd from scans, or generated with
extraction artifacts at the PDF-generation layer — pdftotext output
diverges from the original, and the check verifies the quote matches
a corrupted extract rather than the actual document.

The Grusch PPD-19 OCR case (Phase D) surfaced this failure mode for
ocr-scan sources. But the PPD-19 might be one case among many, or it
might be the only such case. Spot-check confidence is insufficient
to answer the corpus-integrity question: *is every quote in the
corpus actually confirmed against its source, or only against
whatever extraction the pipeline happened to produce?*

Phase F restructured into a per-source audit. For each source,
determine its extraction condition; if the extraction is unreliable,
produce a clean-text sibling; validate existing quotes against the
sibling; surface and correct any findings.

## Schema evolution during the audit

The audit process drives schema changes when new source conditions
surface. Each change is a small design decision, not a retrofit.

**`extraction_type: ocr-scan`** (Phase A, 2026-04-24 earlier). Added
for the Grusch PPD-19 case. Definition: scanned PDF whose text came
from OCR. Failure mode: character-level corruption from the OCR
process itself (e.g. `UAP-telated` for `UAP-related`). Operational
handling: validator prefers same-stem `.txt` sibling.

**`extraction_type: extraction-lossy`** (Tier 1, 2026-04-24). Added
for the hearing transcript PDF case. Definition: text layer is
present and was NOT produced by OCR (author-native or Acrobat
Distiller from a `.txt` source), but pdftotext extraction still
produces artifacts — Unicode mapping errors, font encoding issues,
character substitution at the PDF-generation layer. Different root
cause from ocr-scan (no scan step involved); same operational
consequence (default extraction unreliable). Validator
sibling-lookup generalized from `== "ocr-scan"` to `!= "text-native"`
so the mechanism extends to any non-default extraction_type.

The pattern: binary taxonomies rarely survive first contact with
real data. The Phase A `text-native` vs `ocr-scan` split was right
for Phase A's single case; the hearing transcript surfaced the need
for the third category. Future tiers may surface more.

## Tier 1 — 2023-07-26 House Oversight hearing transcript

Complete 2026-04-24. Commit `4b4d7fb`.

**Source.** `government/congress-gov-house-hearing-transcript-20230726.pdf`.
54 pages, generated by Acrobat Distiller 20.0 from a `.txt` source
(per PDF metadata). 211 quotes cite this source across 6 artifacts —
42% of the corpus.

**Known artifact going in.** nq6 (surfaced Grusch v2, 2026-04-24):
pdftotext renders `11½` (one-half fraction, U+00BD) as `11‡` (double
dagger, U+2021) in Grusch's sworn attestation to "11½ hours" of
classified testimony. Unicode-mapping artifact at the PDF-generation
layer, not an OCR issue.

**Methodology.** Produced a clean-text transcription via VLM page
reading across all 54 pages, preserving government page numbers as
section markers (`=== p. 1 ===` through `=== p. 50 ===`) to match
existing quote Location refs. Flagged the PDF entry
`extraction_type=extraction-lossy` in the manifest. Added the `.txt`
sibling as its own manifest entry with sha256 tracking. Ran the
verbatim-quote check unconditionally against all 211 existing
quotes.

**Findings — 6 corrections across 3 categories:**

### Contributor errors in artifact quote text (3)

| Quote | Correction | Category |
|---|---|---|
| `transcripts/2023-07-26-house-fravor` q3 | dropped extra word "over" — `"observed for over 2 weeks"` → `"observed for 2 weeks"` | drift |
| `transcripts/2023-07-26-house-fravor` q20 | removed fabricated sentence `"No one came."` inserted between real sentences `"battle group."` and `"That captain was aware."` | **content insertion** |
| `transcripts/2023-07-26-house-graves` q12 | added missing "it" — `"drones, is an urgent"` → `"drones, it is an urgent"` | drift |

These errors passed the old "verified verbatim" marker regime
because pdftotext extract of the PDF contained the same contributor-
drifted text they were checked against — there was no independent
reference to surface the divergence. The clean-text sibling IS that
independent reference.

**The q20 "No one came." fabrication is a categorically different
finding** from the other two. Missing/extra words may change
precision but not substantive content. A fabricated sentence, even
a short one, asserts something the witness did not say. `"No one
came."` in context implies an absence-of-response that intensifies
the `"Nothing was done"` framing; the PDF does not include this
sentence. This is the claim-layer failure mode from
`meta/toolkit-notes/pilot-failure-2026-04-17.md` manifesting at
small scale under the verbatim-quote discipline — not just at large
scale in claim-summary form. Preserved here because the distinction
matters for understanding what the audit was worth: the other
errors indicate the transcription workflow needs tighter discipline;
this one indicates a content-insertion failure mode survives even
under "verbatim quote" discipline if no independent reference
checks it.

### Extraction-artifact correction (1)

- `people/david-grusch` q106 + `transcripts/2023-07-26-house-grusch`
  q31 (same text, duplicated across artifacts): `"11‡ hours"` →
  `"11½ hours"`. Known going in (nq6); resolved by Phase D pattern
  (clean text in the `.txt` sibling becomes authoritative; quote
  text updated to match; `naming_quirks` entry nq6 removed because
  preserve-as-sic premise dissolved).

### Errors in the clean-text sibling transcription (2 substantive + 12 typographic-convention)

The same mechanical validation caught errors introduced during
transcription of the `.txt` sibling itself. **This is expected:** the
clean-text workflow is subject to the same transcription failure
mode it's designed to detect. The `.txt` sibling is not a gold
standard by default; it becomes a gold standard by surviving the
same mechanical check everything else is held to.

Substantive (same error class as contributor drift):
- p. 22: `"There are certain colleagues"` → `"There were certain
  colleagues"` (affected Grusch q83 and `transcripts/grusch` q18)
- p. 41: added missing phrase `"the airwing with the carrier"`
  between `"the ships with the carrier,"` and `", and we start
  working"` (affected `transcripts/fravor` q31)

Typographic convention — paired curly singles vs. straight doubles:
- The PDF uses paired curly single-quotes (`‘‘...’’`) as typographic
  substitute for double-quotes throughout (government-print
  tradition; pdftotext preserves as `‘‘...’’`). The initial draft
  used straight doubles (`"..."`) in 12 passages. Two (Graves q1
  `"FOBS"`, Fravor q7 `"sir..."` quote) flagged directly by the
  validator; 10 others corrected proactively for consistency.
  Source-convention discovery, not a contributor error — but worth
  recording as a lesson for future tiers: check pdftotext output
  before assuming typographic conventions.

### Aggregate

- **211 quotes validated** (0 errors, 4 pre-existing word-budget
  warnings in other sources)
- **3 contributor errors corrected** — 1.4% contributor-drift rate
  in previously-"verified" quotes
- **1 known extraction-artifact corrected** (nq6 case, Phase D
  pattern extended)
- **2 substantive + 12 typographic `.txt` transcription errors
  caught by the same validation** — confirming the `.txt`-sibling-
  as-new-gold-standard workflow requires its own mechanical
  check-in, not trust-by-default

### Why Tier 1 was worth doing

The 1.4% contributor-drift rate is a baseline measurement that
didn't exist before this audit. It anchors three claims:

1. **The old "verified verbatim" marker regime had a real failure
   rate** — not catastrophic, but real, and varied in kind (mechanical
   drift vs. content insertion).
2. **A 4-case spot-check on this source would have missed most of
   the findings** — catching one finding out of six in 211 quotes by
   random sampling is <12% probability. The restructured Phase F's
   exhaustive-per-source approach is what caught them.
3. **The clean-text sibling mechanism works as designed** — the
   pattern from Phase D generalized to a new source condition
   (extraction-lossy) without architectural friction, and the
   mechanical validation surfaced all drift categories (contributor
   + transcription) via the same check.

## Tier 2 — written testimony PDFs

Complete 2026-04-24. Commit TBD.

**Sources audited** (87 quotes total across 6 artifacts — 3 document
nodes plus the 3 witness person nodes that duplicate a subset):

| Source | Producer | Pages | Quotes |
|---|---|---:|---:|
| `government/oversight-house-gov-fravor-written-testimony-20230726.pdf` | Acrobat PDFMaker 23 (Word) → Adobe PDF Library 23.3.20 | 3 | 30 |
| `government/docs-house-gov-graves-written-testimony-20230726.pdf` | Skia/PDF m117 (Google Docs Renderer) | 5 | 34 |
| `government/oversight-house-gov-grusch-written-testimony-20230726.pdf` | PScript5.dll → Acrobat Distiller 23.0 (Word) | 3 | 23 |

**Methodology.** Three-step verification per source, total ~10
minutes per PDF:
1. PDF metadata inspection (producer chain reveals the authoring
   pipeline — tells us what to expect from extraction)
2. pdftotext scan for suspect Unicode glyphs (daggers, unusual
   substitutions, fraction / section-sign edge cases) — programmatic
   check
3. Visual verification via VLM page rendering — compare against
   pdftotext output for faithfulness

**Extraction condition.** All three sources: **text-native,
pdftotext faithful.** No extraction artifacts found. No `.txt`
siblings required.

**Findings.** Zero. All 87 quotes currently pass the unconditional
verbatim-quote check (confirmed via pre-commit gates at Tier 2 exit).
Because pdftotext is faithful for each source, the validator's
"pass" is substantively meaningful — these 87 quotes are genuinely
verified against the original document content, not just against a
possibly-corrupted extract. No contributor drift, no fabricated
content, no quote-mark convention issues.

**Notable observation — Distiller behavior varies by content.** The
Grusch written testimony PDF uses the same Acrobat Distiller 23.0
pipeline that produced the hearing transcript PDF (Tier 1's
extraction-lossy case). But the Grusch testimony PDF extracts
cleanly — no `‡` for `½`, no font-encoding artifacts, nothing.
This suggests the hearing-transcript Unicode-mapping failure is
specific to character combinations or document structure in that
particular file, not a blanket property of the Distiller tool.
Practical implication: Distiller-produced PDFs are not
automatically extraction-lossy; individual inspection per source
is what classifies them. The extraction_type enum captures source
CONDITIONS, not production-tool identities.

**Relevance to BACKLOG #21.** The "alternate PDF extraction tool"
hypothesis (#21) was framed around the hearing transcript's
Unicode-mapping issue. Tier 2 provides a useful data point: most
Distiller PDFs don't need alternate tooling. If a second
extraction-lossy case surfaces in Tier 5 (PDF long-tail), #21's
evaluation becomes more valuable; if not, #21 stays deferred as a
single-case fix no other corpus source would benefit from.

**Audit time.** ~30 minutes across all three sources. Significantly
faster than Tier 1 (which required a full 54-page transcription).
Tier 2's speed is the baseline for text-native audits going forward.

## Tier 3 — transcript sources (pending; convention decision blocks audit)

Targets: 9 sources, 73 quotes. Mostly YouTube-downloaded auto-
captions (the `-downloaded.md` pattern in `sources/transcripts/`):
JRE 2065 (20), NewsNation Coulthart (17), American Alchemy (10),
Somewhere in the Skies (9), Grusch Sol closing (6), Linda Hall (5),
American Veterans Center (3), Merged Podcast (2), Calling All
Beings (1). One PDF-format transcript (When It Mattered, 6 quotes)
— worth categorizing whether it's a human-produced transcript or a
caption re-packaging.

**Convention decision required before audit execution.** Phase E's
principle ("confirmation is a precondition for inclusion, against
the underlying primary source") doesn't directly specify how to
handle audio sources. Three framings:

1. **Transcript-as-source.** What Phase E language literally says.
   Under this rule, Tier 3 audit just checks quotes match transcript
   file, which the validator already does. Clean but accepts
   caption errors as authentic to the source.
2. **Audio-as-source.** Every transcript-derived quote needs
   confirmation against audio. Substantial work (download + sample-
   verify all 9 sources). Possibly automate via modern speech-to-
   text + diff-against-caption, human-verify disagreements.
3. **Hybrid by transcript provenance.** Equivalent-footing when
   human-produced (stenographic transcripts, published interview
   transcripts — human has already done confirmation-against-audio
   work). Audio-confirmation required when auto-caption (machine
   extraction of audio; same shape of problem as OCR vs. page
   image). Requires classifying each of the 9 sources.

Option 3 is most principled; requires a `transcript_provenance`
field on transcript manifest entries (or similar classifier).
Decision lives in `meta/conventions.md` alongside the Phase E
principle.

## Running methodology log

Observations worth preserving across tiers, accumulated as each tier
executes:

**From Tier 1:**
- Check pdftotext output before assuming typographic conventions on
  a PDF source. Government-print tradition uses paired curly singles
  for double-quotes; pdftotext preserves this exactly. Straight
  doubles in a transcription diverge from the source.
- The clean-text transcription workflow is subject to the same
  transcription failure mode it's designed to detect. Run the
  mechanical check before declaring the `.txt` sibling authoritative.
- Fabricated content (as distinct from mechanical drift) can
  survive the verbatim-quote discipline if no independent reference
  is checking. The mechanical backstop is what closes that gap.
- Binary schema taxonomies rarely survive first contact with real
  data. Expect the extraction_type enum to grow as new tiers
  surface new conditions. Each addition is a small design decision,
  not a retrofit.

**From Tier 2:**
- Text-native PDFs from standard word-processor pipelines (Word
  + Adobe PDFMaker, Word + Acrobat Distiller for short docs,
  Google Docs Skia) produce faithful pdftotext output. The
  existing unconditional verbatim-quote check is substantively
  meaningful when pdftotext is faithful — no `.txt` sibling
  needed.
- Producer-tool identity doesn't classify extraction condition on
  its own. Grusch written testimony = same Acrobat Distiller 23.0
  as the hearing-transcript extraction-lossy case, but extracts
  cleanly. Per-source inspection is what classifies.
- Three-step verification (metadata → pdftotext suspect-char scan →
  visual VLM check) is the right lightweight methodology for text-
  native PDFs. ~10 minutes per source. Scales to Tiers 4 and 5.
- "Clean pass" audits have real value — they convert the validator's
  "quote matches pdftotext" result from "possibly meaningful" to
  "substantively meaningful" once pdftotext is verified faithful.
  Recording the verification is what makes the audit state
  durable.
